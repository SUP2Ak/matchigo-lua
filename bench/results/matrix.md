# matchigo-lua bench matrix

Generated by `bench/matrix.lua` from per-runtime stats sidecars under
`bench/results/.stats/`. Each runtime is rebuilt independently — to
update one column, just rerun `bench/run.lua` under that interpreter
and re-run `lua bench/matrix.lua`.

> [!IMPORTANT]
> **Reading these numbers in context.** Microbenches measure the
> *abstraction's* per-call overhead in tight loops with cycling
> inputs — that's the point, not a flaw.
>
> In most production code, Lua compute runs in **nanoseconds**
> while the work around it (a SQL query, an HTTP call, a file
> read, a queue pop) runs in **milliseconds**. A `2× slower` row
> here is ~50–200 ns extra per dispatch — orders of magnitude
> below the noise floor of any external IO.
>
> **Where it does matter** : real-time framing contexts (game
> engines like LÖVE/Defold, ~16 ms frame budget at 60 Hz),
> high-frequency event loops (OpenResty under load), or any tight
> inner loop with no IO. In those, **the alloc column matters
> more than the ns column** — sustained per-call allocation
> causes GC pauses that *will* eat your frame budget. Hot paths
> with `alloc = 0 B` are GC-safe regardless of which dispatcher
> you pick.
>
> Pick on readability + maintainability + your runtime's actual
> allocation pressure. The bench tells you what dispatch costs ;
> only your profiler tells you whether it matters for *your* app.

> [!NOTE]
> **Reading LuaJIT cells** : ratios near 1.0× and small absolute ns
> values are not measurement failures. Cycling inputs defeat constant
> folding, but the JIT can still inline both contestants down to
> nearly identical machine code on pure dispatches. Under JIT,
> **allocation is the one cost the optimizer cannot make disappear**
> — that's the column to watch.

## Summary

Median ratios across **real-world** scenarios per runtime
(scenarios named `educational` are excluded from the medians).
`compile()` = `m.compile(rules)` data-driven dispatch.
`matcher+DSL` = `m.matcher():with(...)` chained API with the
Rust-style DSL. The **alloc-neutral** columns count scenarios
where matchigo's hot path allocates the same bytes/call as
native (within 8 B noise) — i.e., matchigo adds no allocation
pressure that wasn't already there in the equivalent hand-written
code.

### JIT folding showcase — constant input (educational)

> [!TIP]
> **Educational — read the LuaJIT column carefully.** This bench
> deliberately uses a **constant** input every iteration (no
> cycling). On standard Lua you'll see the real per-call cost ;
> on LuaJIT both contestants collapse to ~0 ns because the JIT
> inlines the body and hoists the compute out of the loop
> (LICM). The lesson : **in a hot loop with a constant input,
> LuaJIT erases dispatch overhead entirely**, regardless of which
> dispatcher you pick. Compare against the cycled scenarios above
> for the actual dispatch cost when the JIT cannot fold.

| benchmark | `5.3` | `5.4` | `luajit-2.1` |
| --- | ---: | ---: | ---: |
| `native       if/elseif (constant 'POST')` | 37 ns · 0 B · _(base)_ | 40 ns · 0 B · _(base)_ | 0 ns · 0 B · _(base)_ |
